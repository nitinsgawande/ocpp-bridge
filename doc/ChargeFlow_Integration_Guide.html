Integrate postpaid
EV charging

ChargeFlow is a UEI Beckn BPP that enables postpaid metered EV charging sessions — the driver scans a QR, charges, and pays the exact kWh amount after the session ends via UPI collect. No wallet, no pre-payment, no app download required.

Payment model

Payment type ON-FULFILLMENT per UEI spec PR #308. Driver is charged the verified kWh amount after session ends — not before. Cable lock enforces payment before connector releases.

Architecture

ChargeFlow sits between your BAP app and the CPO's CSMS — handling the OCPP protocol, session metering, and UPI collect flow transparently.

Data flow
Your App (BAP)
    │  Beckn search/confirm/status
    ▼
ChargeFlow BPP  ──── Redis (session state)
    │               MongoDB (audit log)
    │  CHARGEFLOW_ idTag + webhook
    ▼
CPO CSMS (Pulse Energy / Kazam)
    │  OCPP 1.6J
    ▼
Physical Charger (CCS2 + cable lock)
    │  After payment confirmed
    └── UnlockConnector → cable releases

CPO side — configuration only

The CPO makes zero code changes. They configure one webhook routing rule in their CSMS admin panel:

CPO CSMS configuration
# If idTag starts with CHARGEFLOW_
# POST session events to ChargeFlow webhook

Condition : idTag.startsWith("CHARGEFLOW_")
Action    : POST to https://api.chargeflow.in/webhook/cpo/session
Events    : StartTransaction · MeterValues · StopTransaction

Quick Start

1 — Register your BAP

Contact ChargeFlow to register your BAP ID and URI. You will receive a BPP endpoint and your BAP credentials.

VariableDescriptionExample
BAP_IDYour BAP identifierpulse-energy.in
BAP_URIYour callback base URLhttps://api.pulseenergy.in
BPP_IDChargeFlow BPP IDchargeflow.in
BPP_URIChargeFlow endpointhttps://api.chargeflow.in

2 — Set up your callback endpoint

ChargeFlow sends all responses asynchronously to your BAP URI. You must expose these routes:

BAP callback endpoints to implement
POST /on_search   ← charger catalogue
POST /on_select   ← tariff breakdown
POST /on_init     ← order draft
POST /on_confirm  ← session started
POST /on_status   ← live cost (every 30s) + final
POST /on_update   ← session receipt

3 — Complete a test session

Import the Postman collection and run the requests in order: search → select → init → confirm → status → update. Each returns an ACK immediately — check your BAP callback logs for the actual response data.

Beckn async pattern

Every endpoint returns {"message":{"ack":{"status":"ACK"}}} immediately. The actual response data is sent to your BAP_URI callback. This is mandatory in the Beckn protocol — never wait for data in the initial response.

Beckn BPP Endpoints

POST /beckn/search Find chargers near location ▼

Search for ON-FULFILLMENT capable DC fast chargers near a GPS location. Include payment.type: "ON-FULFILLMENT" in intent to filter postpaid-only chargers.

Request body 
{
  "context": {
    "domain": "ev-charging:uei",
    "action": "search",
    "bap_id": "YOUR_BAP_ID",
    "bap_uri": "https://your-bap.com",
    "transaction_id": "txn-001"
  },
  "message": {
    "intent": {
      "fulfillment": {
        "end": { "location": { "gps": "12.9716,77.5946" } }
      },
      "payment": { "type": "ON-FULFILLMENT" }
    }
  }
}

Callback: ChargeFlow sends on_search to your BAP_URI with charger catalogue including connector type, power rating, tariff, and location.

POST /beckn/confirm ★ Confirm postpaid session ▼

The key action. Driver confirms the postpaid session. ChargeFlow calls the CPO API to start the charging session.

Critical — must use ON-FULFILLMENT

payments[0].type must be "ON-FULFILLMENT" (with hyphen). If you send "PRE-FULFILLMENT" or "ON_FULFILLMENT" the request is rejected.

Request body — payments object 
"payments": [
  {
    "type":         "ON-FULFILLMENT",  ← PR #308 — postpaid
    "status":       "NOT-PAID",        ← driver has not paid yet
    "collected_by": "BPP",             ← ChargeFlow collects
    "params": {
      "vpa":      "driver@ybl",      ← driver UPI ID
      "currency": "INR"
    }
  }
]

What happens after confirm

1
ChargeFlow → CPO API: RemoteStartTransaction
idTag = CHARGEFLOW_{driver_phone}
2
CPO CSMS → Charger: session starts
Physical charger begins dispensing energy
3
ChargeFlow → BAP: on_confirm (ACTIVE)
Session confirmed, order ID returned
4
ChargeFlow → BAP: on_status every 30s
LIVE-METER tag: kWh consumed + cost so far
POST /beckn/status Live cost · session state ▼

Poll session status. ChargeFlow also pushes on_status automatically every 30 seconds — you only need to call this for on-demand checks.

BAP Callbacks

ChargeFlow calls these endpoints on your BAP_URI asynchronously. Implement all six to handle the complete session lifecycle.

CallbackWhen sentKey data
POST /on_searchAfter /searchCharger catalogue with tariffs
POST /on_selectAfter /selectTariff breakdown + GST
POST /on_initAfter /initOrder draft + payment terms
POST /on_confirmAfter /confirmSession ACTIVE + order ID
POST /on_statusEvery 30s during session + on paymentLIVE-METER tag · SESSION-SUMMARY tag
POST /on_updateAfter /updateFinal receipt + PAID status
Your callback must return 200 OK

ChargeFlow expects a 200 response from your callback within 10 seconds. If your endpoint is unreachable, on_status callbacks for live cost will be dropped for that interval — the session continues normally.
